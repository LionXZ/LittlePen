# 长期优化方案：数学批改 + RAG 科学批改

> 目标：跨学科扩展，技术探索性质。预计 1-2 月，先 POC 再量产。

---

## 1. 数学批改

### 1.1 核心难点

数学批改和作文批改的本质差异：

```
作文批改: 主观 → 大模型判断质量
数学批改: 客观 → 对就是对，错就是错，不能模棱两可
```

主要挑战：
1. 大模型本身会算错（LLM 不是计算器）
2. 解题步骤判断需要逻辑推理
3. 口语/竖式/分数等格式 OCR 识别率低

### 1.2 两阶段方案

```
Step 1: 题目提取
  图片 → OCR 模型提取"题目区域"和"答案区域"

Step 2: 批改判断
  有标准答案 → 直接对照  + deepseek 分析解题步骤是否完整
  无标准答案 → deepseek 自行求解 → 对比答案 → 分析差异
```

### 1.3 参考答案字段

二维码新增第 8 段 `answer_hash`（可选）：

```
课程ID-班级ID-排课ID-学号-姓名-性别-科目-答案哈希
```

`answer_hash` 不是答案内容，而是答案版本号（如 `v1`、`v2`），后端通过它查询对应标准答案：

```python
# 答案配置表
ANSWER_KEY = {
    "math-grade3-v1": {"1": "42", "2": "108", "3": "15.6"},
    "math-grade3-v2": {"1": "36", "2": "99", "3": "12.3"},
}
```

### 1.4 数学批改流水线

和作文批改不同的流水线结构：

```
START → qr_parse → math_ocr → [answer_check | step_analysis] → aggregate → END
                                        (并行)
```

```
graph.py 中新增分支:
  if subject == "ma":
      workflow.add_edge("qr_parse", "math_ocr")
      workflow.add_edge("math_ocr", "answer_check")
      workflow.add_edge("math_ocr", "step_analysis")
      ...
  else:
      走原有作文流程
```

### 1.5 新增 Tool

| Tool | 功能 | 调用模型 |
|------|------|---------|
| `math_ocr_tool` | 识别算式、数字、竖式、分数 | GLM-5V-Turbo（多模态），加 math-focused prompt |
| `answer_check_tool` | 对比标准答案，判断对错 | 纯逻辑，不调模型 |
| `step_analysis_tool` | 分析学生解题步骤是否合理 | deepseek-v4-pro |

### 1.6 数学评分维度

| 维度 | 说明 | 分值 |
|------|------|------|
| 正确性 | 最终答案是否正确 | 0-50 分 |
| 解题步骤 | 步骤是否完整、逻辑清晰 | 0-25 分 |
| 卷面整洁 | 书写规范性 | 0-15 分 |
| 规范性 | 单位、格式是否正确 | 0-10 分 |

### 1.7 技术风险

| 风险 | 缓解措施 |
|------|---------|
| LLM 自己算错 | 标准答案模式下不依赖 LLM 计算能力，只做对比 |
| 竖式/分数 OCR 不准 | 先用最简单的横式算式做 POC，逐步扩展 |
| 解题步骤判断主观 | 提供参考答案中的标准步骤作为对比基准 |

### 1.8 文件改动

| 文件 | 改动 |
|------|------|
| `tools/math_ocr_tool.py` | **新文件** — 数学 OCR |
| `tools/math_check_tool.py` | **新文件** — 答案对照 |
| `tools/step_analysis_tool.py` | **新文件** — 步骤分析 |

## 2. RAG 知识库辅助科学批改

### 2.1 场景

物理/化学/生物的填空题、简答题。这些题目没有标准模板化的答案，需要判断学生回答是否"触及知识点"。

```
题目: "为什么铁会生锈？"
学生 A: "因为铁和氧气反应"           → 部分正确
学生 B: "铁和水和氧气发生化学反应"     → 正确
学生 C: "因为铁碰到了水"               → 不够准确
```

要求学生答案中包含"氧气"和"水"两个关键词，且表述了"化学反应"概念。

### 2.2 方案

```
教材/教案 PDF
    │
    ▼ 向量化
┌──────────────┐
│  ChromaDB    │  ← 向量数据库（本地部署）
│  知识点向量库  │
└──────┬───────┘
       │
批改时: 学生答案 → 向量检索 → 找最相关的知识点
       │
       ▼
  LLM 判断: 学生答案是否覆盖了这个知识点的核心内容
```

### 2.3 知识库构建流程

```
1. 收集教材/教案 → 分块 (chunk_size=500)
2. Embedding: sentence-transformers 把文本转成向量
3. 存入 ChromaDB
4. 定期更新（新增教材时重新入库）
```

### 2.4 批改 Prompt 示例

```
题目: {question}
参考答案涉及知识点: {retrieved_knowledge}

学生答案: {student_answer}

请判断学生答案是否包含了知识点的核心内容：
1. 完全正确（涵盖所有关键概念）
2. 部分正确（提到了部分概念但不够完整）
3. 不正确（关键概念缺失或理解错误）

给出简短评语，指出缺失的知识点。
```

### 2.5 新增 Tool

| Tool | 功能 |
|------|------|
| `rag_retrieve_tool` | 根据学生答案从 ChromaDB 检索相关知识点 |
| `knowledge_check_tool` | 判断答案与知识点的匹配程度 |

### 2.6 新增依赖

```txt
chromadb>=0.5
sentence-transformers>=3.0
```

### 2.7 docker-compose

```yaml
# ChromaDB 不需要额外容器，本地文件存储即可
# chromadb 默认以嵌入模式运行
```

### 2.8 局限与风险

| 风险 | 说明 |
|------|------|
| 知识库构建成本 | 需要人工整理教材/教案，不能自动生成 |
| 检索精度 | Embedding 对中文学科术语的向量表示效果需要验证 |
| 覆盖度 | 只能批改"有知识点支撑"的内容，开放性问题无法处理 |

## 3. 不纳入规划的内容

以下是有意不做的：

| 方向 | 原因 |
|------|------|
| 语音批改 | 技术链太长（ASR→批改→TTS），受众小 |
| 实时互动式批改 | 当前异步批改流程已满足需求，实时交互需要 WebSocket 改架构 |
| AI 出题 | 和批改是两个产品方向，不在此路线图中 |
| 移动端 App | Web 端优先，移动端可用 PWA 或微信小程序（后期考虑） |

---

## 4. 优先级

```
数学批改 > RAG 科学批改

原因：
  数学批改有明确的对错标准，技术路线清晰
  RAG 依赖知识库积累和向量检索，前期基建工作量大
```
